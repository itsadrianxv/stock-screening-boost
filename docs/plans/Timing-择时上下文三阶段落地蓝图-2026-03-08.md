# Timing 择时上下文三阶段落地蓝图（2026-03-08）

## 1. 文档目标

本文档用于将 `timing` 择时上下文规划为一个可分阶段落地、可直接交由 coding agent 执行的工程方案。本文档重点回答以下问题：

1. `timing` 上下文在本项目中的职责边界是什么。
2. 它与现有 `screening / intelligence / workflow / watchlist` 如何协作。
3. 采用怎样的多 agent 结构，才能在保持投研平台定位的前提下，为自选股提供交易建议。
4. 三个阶段分别应实现什么、暂缓什么、如何验收。
5. coding agent 在实现时不应再自行拍板的关键接口、数据模型、工作流接入点与默认行为是什么。

本文档默认遵守当前仓库的技术与架构约束：

- 前端与业务编排：Next.js App Router + TypeScript + tRPC + Prisma
- 金融数据网关：Python FastAPI
- 分层架构：DDD（domain / application / infrastructure）
- 工作流编排：LangGraph.js
- UI 定位：投研平台，不直接承担自动交易执行职责

---

## 2. 结论先行：`timing` 的产品定位

`timing` 不应在第一版被定义为“完整交易系统”，而应被定义为：

> 一个基于确定性技术信号、多 agent 解释与组合建议机制，为单股、自选股列表、筛选结果提供“择时建议”的研究上下文。

换句话说：

- **不是** 自动下单器
- **不是** 券商级风险系统
- **不是** 完整回测引擎
- **而是** 面向研究与决策辅助的“信号 -> 风险 -> 建议 -> 复查”闭环

因此，`portfolio_manager` 与 `risk_manager` 的思路可以借鉴，但语义需要调整：

- `watchlist_risk_manager`：输出风险预算、暴露上限、禁做条件、组合拥挤提示
- `watchlist_portfolio_manager`：输出交易建议、优先级、建议仓位区间、操作顺序

其输出应始终是“建议”，而不是“执行指令”。

## 3. 与现有上下文的关系

### 3.1 `timing` 与 `screening`

- `screening` 负责“选出值得研究/观察的候选池”
- `timing` 负责“判断这些候选池里哪些更接近当前可观察、试仓、加仓、减仓或退出的时点”

推荐的串联关系：

- `screening` 先做横截面筛选
- `timing` 再做时序维度判断

阶段三默认联动方式写死如下：

- `screening -> timing` 自动触发挂在 `ScreeningExecutionService` 完成 session 后
- 挂点与当前 `screening insight pipeline` 的 dispatch 位于同一位置并行触发
- `screening_to_timing_v1` 默认从 `ScreeningSession.topStocks` 中截取前 `20` 个候选进入 timing

### 3.2 `timing` 与 `intelligence`

- `intelligence` 负责证据、主题、新闻、公司基本面解释
- `timing` 负责技术信号、市场状态和交易建议

后续可逐步把 `intelligence` 中的新闻/事件能力叠加给 `timing`，形成 `event_risk_agent`。但在当前蓝图中：

- 阶段一与阶段二不依赖 `intelligence` 的新闻能力
- `market_regime_agent` v1 只消费规则化市场快照，不叠加新闻或事件

### 3.3 `timing` 与 `watchlist`

- `watchlist` 是 `timing` 的天然目标池
- `timing` 应支持“针对单个 watchlist 批量生成 timing cards”
- 第二阶段开始，应让 `watchlist` 成为 `timing` 的第一主要入口，而不是仅支持单股查询

### 3.4 `timing` 与 `workflow`

`timing` 不应另起一套调度体系，而应作为新的 Workflow Template 挂入现有 LangGraph 体系。模板划分冻结如下：

- 阶段一：
  - `timing_signal_pipeline_v1`：单股择时卡
  - `watchlist_timing_cards_pipeline_v1`：watchlist 批量择时卡
- 阶段二：
  - `watchlist_timing_pipeline_v1`：在批量择时卡基础上追加市场状态、风险预算与组合建议
- 阶段三：
  - `screening_to_timing_v1`
  - `timing_review_loop_v1`

`timing` 的写入口统一走现有工作流基础设施，而不是绕开 workflow 单独发起同步任务。

## 4. 核心设计原则

### 4.1 计算与解释分离

- **Python**：负责价格序列获取与确定性指标计算
- **TypeScript + LangGraph**：负责工作流编排、状态持久化、LLM 解释、多 agent 合成
- **LLM**：只负责解释、归纳、排序、提示，不负责原始技术指标计算

### 4.2 先建议、后组合、再闭环

产品推进顺序应为：

1. 单股/批量择时卡
2. 自选股交易建议
3. 复盘与复查闭环

### 4.3 先日频，再考虑更高频

第一版统一采用**日频**数据：

- 日线 OHLCV
- 日级技术指标
- 日级组合建议

不在第一版引入：

- 分钟级信号
- 盘中提醒
- 实时流式行情
- 自动下单

### 4.4 先可解释，再追求复杂度

首批指标只做高共识、低争议、易解释的集合：

- MACD
- RSI
- Bollinger Bands
- OBV
- EMA20 / EMA60
- ATR14
- Volume Ratio 20D

### 4.5 规则先于 LLM

- 原始指标数值必须由确定性逻辑产出
- 风险预算、仓位上限、禁做条件必须由规则或确定性计算产出
- LLM 只能填充 schema 允许的枚举、解释、摘要与排序字段
- 任一节点输出在持久化前都应通过固定 schema 校验

## 5. 实现决策冻结

本节用于冻结 coding agent 不应再自行决定的关键实现策略。

### 5.1 执行模式冻结

- `timing` 的写入口统一采用“异步 workflow run + worker + run 页面”模式
- 阶段一不是同步 `generate*` 返回完整结果，而是 enqueue workflow、返回 `runId`
- 用户通过 run 页面查看执行过程；`/timing` 页面与 `timingRouter` 只读取持久化结果
- `workflowRouter` 负责启动 pipeline；`timingRouter` 只负责查询与 CRUD

### 5.2 Python 契约冻结

- `timing` 所有 Python v1 接口统一复用 `GatewayResponse / GatewayErrorResponse`
- 单标的接口失败时直接返回标准错误体
- 批量接口采用 `items[] + errors[]` 的部分失败模式
- 不引入 timing 专属的裸 JSON 响应风格

### 5.3 Prisma 关系策略冻结

- 采用 `Hybrid`：timing 核心表自成体系，但对高价值上下文保留显式关系
- `sourceType/sourceId` 继续保留，用于表达来源语义，不替代显式外键
- `PortfolioSnapshot` 属于 timing 自有聚合，不挂到 screening 上下文

### 5.4 节点 I/O 冻结

- 所有 workflow 节点输出固定结构化 JSON
- 不允许以自然语言段落代替节点核心输出
- 节点 schema 应满足：可恢复执行、可落表、可测试、可比较版本差异

### 5.5 阶段三复用策略冻结

- `screening -> timing` 复用现有 workflow 链路
- reminder 基础设施不另起一套，但现有 `ResearchReminder` 需要抽象为可挂多目标的通用提醒模型
- 阶段三不新建独立 `TimingReminder` 表

## 6. 建议的领域模型总览

建议在 `src/server/domain/timing/` 下建立独立上下文，并按 DDD 分层组织。

### 6.1 聚合根

1. `TimingAnalysisCard`
- 面向用户展示的择时分析结果
- 聚合单股或单个标的的完整建议视图

2. `TimingRecommendation`
- 面向自选股组合建议的结果对象
- 聚合风险预算、动作建议、优先级等信息

3. `PortfolioSnapshot`
- 用户自定义的组合快照
- 作为 `watchlist_risk_manager` / `watchlist_portfolio_manager` 的输入基础

4. `TimingReviewRecord`
- 面向复查闭环的结果对象
- 聚合择时建议的预期、结果、偏差与复盘结论

### 6.2 值对象与枚举

1. `TechnicalSignalSet`
- 存储 MACD、RSI、OBV、Bollinger、EMA、ATR 等快照

2. `MarketRegime`
- `RISK_ON | NEUTRAL | RISK_OFF`

3. `TimingAction`
- `WATCH | PROBE | ADD | HOLD | TRIM | EXIT`

4. `RiskFlag`
- 如 `HIGH_VOLATILITY`、`TREND_WEAKENING`、`OVERBOUGHT`、`CROWDING_RISK`、`HIGH_CORRELATION`

5. `InvalidationRule`
- 描述“本建议在什么条件下失效”

6. `ReminderTargetType`
- `SCREENING_INSIGHT | TIMING_REVIEW`

### 6.3 领域服务

1. `TimingSignalPolicy`
- 把多个技术指标映射到统一的方向性信号

2. `TimingConfidencePolicy`
- 根据指标一致性、市场状态等生成置信度

3. `PortfolioSuggestionPolicy`
- 在风险约束内输出优先级与动作建议

4. `TimingReviewPolicy`
- 在既定观察周期内评估建议结果、失效条件与复查结论

## 7. 建议的数据持久化模型

本节是 Prisma 设计草案。字段可在编码阶段微调，但以下关系策略、字段职责与默认语义不应偏离。

### 7.1 关系策略总则

- timing 核心结果使用独立表，不直接复用 screening/intelligence 的结果表
- 对运行态、watchlist、portfolio 等强关联上下文使用显式 relation
- 对来源语义保留 `sourceType/sourceId`
- workflow 产物默认允许追溯到 `workflowRunId`

### 7.2 阶段一必须有

#### `TimingSignalSnapshot`

- `id`
- `userId`
- `workflowRunId?`
- `stockCode`
- `stockName`
- `asOfDate`
- `sourceType`：`single | watchlist | screening`
- `sourceId`
- `timeframe`：固定 `DAILY`
- `barsCount`
- `indicators`：JSON，存确定性指标快照
- `signalSummary`：JSON，存规则级方向摘要
- `createdAt`

说明：

- `workflowRunId?` 用于回溯是哪次 pipeline 产出的快照
- `sourceType/sourceId` 仅表达“这张快照来源于哪里”，不承担强关系约束职责

#### `TimingAnalysisCard`

- `id`
- `userId`
- `workflowRunId?`
- `watchListId?`
- `stockCode`
- `stockName`
- `sourceType`
- `sourceId`
- `signalSnapshotId`
- `actionBias`：`WATCH | PROBE | ADD | HOLD | TRIM | EXIT`
- `confidence`
- `marketRegime?`
- `summary`
- `triggerNotes`
- `invalidationNotes`
- `riskFlags`：字符串数组
- `reasoning`：JSON，保存结构化解释
- `createdAt`
- `updatedAt`

阶段一默认行为冻结如下：

- 当不存在 `PortfolioSnapshot` 或持仓上下文时，`actionBias` 只允许产出 `WATCH | PROBE | ADD`
- `HOLD | TRIM | EXIT` 仅在存在持仓上下文或第二阶段组合建议流程中允许出现

### 7.3 阶段二建议新增

#### `PortfolioSnapshot`

- `id`
- `userId`
- `name`
- `baseCurrency`
- `cash`
- `totalCapital`
- `positions`：JSON，包含标的、数量、成本、当前权重
- `riskPreferences`：JSON，包含最大单票、最大主题暴露、默认试仓比例等
- `createdAt`
- `updatedAt`

说明：

- `PortfolioSnapshot` 是 timing 自有聚合
- 它不从属于 `screening`，也不通过 `screeningSessionId` 建模

#### `TimingRecommendation`

- `id`
- `userId`
- `workflowRunId?`
- `portfolioSnapshotId`
- `watchListId`
- `stockCode`
- `stockName`
- `action`
- `priority`
- `confidence`
- `suggestedMinPct`
- `suggestedMaxPct`
- `riskBudgetPct`
- `marketRegime`
- `riskFlags`
- `reasoning`
- `createdAt`
- `updatedAt`

### 7.4 阶段三建议新增

#### `TimingReviewRecord`

- `id`
- `userId`
- `analysisCardId?`
- `recommendationId?`
- `reviewHorizon`：`T5 | T10 | T20`
- `scheduledAt`
- `completedAt?`
- `expectedAction`
- `actualReturnPct?`
- `maxFavorableExcursionPct?`
- `maxAdverseExcursionPct?`
- `verdict?`
- `reviewSummary?`
- `createdAt`
- `updatedAt`

说明：

- 阶段三的 review loop 以 `scheduledAt` 为到期依据
- `analysisCardId` 与 `recommendationId` 允许二选一或同时存在，具体由来源决定

#### `TimingPreset`

- `id`
- `userId`
- `name`
- `description`
- `config`：JSON，包含指标权重、agent 权重、阈值配置
- `createdAt`
- `updatedAt`

#### `ResearchReminder` 泛化策略

阶段三不新建独立 reminder 表，而是泛化现有 `ResearchReminder`：

- 增加 `targetType`：`SCREENING_INSIGHT | TIMING_REVIEW`
- 将当前强绑定 `insightId` 升级为：
  - `screeningInsightId?`
  - `timingReviewRecordId?`
- 在应用层与领域层强校验“两个目标字段必须且只能有一个”

## 8. Python FastAPI 侧落地蓝图

### 8.1 阶段一目标

为 `timing` 提供最基础的日频行情与指标能力。

### 8.2 建议新增目录/文件

- `python_services/app/contracts/timing.py`
- `python_services/app/services/timing_indicators.py`
- `python_services/app/gateway/timing_gateway.py`
- `python_services/app/routers/timing_v1.py`

### 8.3 响应契约冻结

所有 `timing` v1 接口统一复用现有标准化 contract：

- 成功：`GatewayResponse<T>`
- 失败：`GatewayErrorResponse`

单标的接口：

- `GatewayResponse<TimingBarsData>`
- `GatewayResponse<TimingSignalData>`

批量接口：

- `GatewayResponse<TimingSignalBatchData>`
- 其中 `TimingSignalBatchData = { items, errors }`

### 8.4 阶段一接口规范

#### 1. `GET /api/v1/timing/stocks/{stockCode}/bars`

输入：

- `start`：可选
- `end`：可选
- `timeframe`：固定只接受 `DAILY`，缺省为 `DAILY`
- `adjust`：可选，缺省为 `qfq`

默认行为：

- 当 `end` 未传时，使用 Python 服务可获得的最新交易日
- 当 `start` 未传时，由服务按内部最小 lookback 要求回溯足够数据

输出 `TimingBarsData` 至少包含：

- `stockCode`
- `stockName`
- `timeframe`
- `adjust`
- `bars[]`：标准化 OHLCV 日线数组

#### 2. `GET /api/v1/timing/stocks/{stockCode}/signals`

输入：

- `asOfDate`：可选，缺省取最新交易日
- `lookbackDays`：可选，提供指标计算所需最小回溯长度

输出 `TimingSignalData` 至少包含：

- `stockCode`
- `stockName`
- `asOfDate`
- `barsCount`
- `indicators`
  - `macd`
  - `rsi`
  - `bollinger`
  - `obv`
  - `ema20`
  - `ema60`
  - `atr14`
  - `volumeRatio20`
- `ruleSummary`
  - `direction`
  - `signalStrength`
  - `warnings[]`

说明：

- `ruleSummary` 只允许规则计算，不允许 LLM 参与

#### 3. `POST /api/v1/timing/stocks/signals/batch`

输入：

- `stockCodes[]`
- `asOfDate`：可选，缺省取最新交易日
- `lookbackDays`：可选

输出 `TimingSignalBatchData`：

- `items[]`：成功返回的信号快照
- `errors[]`：逐股票错误

部分失败策略冻结如下：

- 单只股票失败不应导致整个 batch 失败
- 只有请求体整体非法、上游整体不可用时才返回标准错误体

### 8.5 阶段二接口规范

#### 4. `GET /api/v1/timing/market/regime-snapshot`

用途：

- 为 `market_regime_agent` 提供纯规则化输入

输出至少包含：

- 指数级别快照
- 市场宽度指标
- 波动相关指标
- 规则归纳所需的中间特征

约束：

- 不引入新闻
- 不引入 LLM
- 不在 Python 侧直接产出最终组合建议

### 8.6 Python 侧职责边界

Python 侧只做：

- 拉取日线数据
- 清洗数据
- 计算指标
- 生成规则级摘要
- 生成市场状态计算所需的规则化快照

Python 侧不做：

- LLM 解释
- 组合建议
- 工作流编排
- Prisma 持久化

## 10. 多 agent 设计建议

本节冻结各 agent 的最小输入/输出 schema。实现时可扩展字段，但不得减少以下核心字段。

### 10.1 第一阶段 agent

#### `technical_signal_agent`

输入：

- `signalSnapshots[]`

其中每项至少包含：

- `stockCode`
- `stockName`
- `asOfDate`
- `barsCount`
- `indicators`
- `ruleSummary`

输出：`technicalAssessments[]`

其中每项至少包含：

- `stockCode`
- `direction`：`bullish | neutral | bearish`
- `signalStrength`
- `confidence`
- `factorBreakdown`
- `triggerNotes`
- `invalidationNotes`
- `riskFlags`
- `explanation`

说明：

- 该 agent 不负责计算指标
- 该 agent 只负责解释“这些指标组合起来代表什么”
- 输出必须是结构化 JSON，不允许只返回纯文本分析

#### `timing_synthesis_agent`

输入：

- `technicalAssessments[]`

## 14. 三个阶段的详细实施计划

## 14.1 阶段一：可信的信号工厂 + 择时卡 MVP

### 目标

建立 `timing` 的最小闭环：

- 能拿到日线数据
- 能计算确定性技术指标
- 能输出单股与 watchlist 的择时卡
- 能通过 workflow run 页面观察执行过程

### 必须完成

1. Python 日线 OHLCV 接口
2. Python 技术指标计算接口
3. Python 批量信号接口（支持部分失败）
4. TS 侧 timing data client
5. `timing_signal_pipeline_v1`
6. `watchlist_timing_cards_pipeline_v1`
7. `TimingSignalSnapshot` / `TimingAnalysisCard` 持久化
8. `/timing` 页面 MVP
9. `workflowRouter` 的两个 start 入口

### 明确不做

- 组合建议
- 风险预算
- 相关性分析
- review loop
- 回测

### 验收标准

1. 用户可以对单个股票启动 timing pipeline 并获得 `runId`
2. 用户可以对一个 watchlist 启动批量 timing pipeline 并获得 `runId`
3. 每张 card 至少包含：
   - `actionBias`
   - `confidence`
   - `summary`
   - `triggerNotes`
   - `invalidationNotes`
   - 关键指标摘要
4. 当无持仓上下文时，`actionBias` 不得出现 `HOLD | TRIM | EXIT`
5. 任一数据拉取失败时，系统应返回可解释错误，而不是静默失败

## 14.2 阶段二：自选股交易建议中心

## 14.3 阶段三：闭环复查与跨上下文联动

## 15. 编码约束、默认行为与测试建议

### 15.1 命名建议

- 上下文统一使用 `timing`
- 模板码统一使用 `*_v1`
- 风险与组合建议 agent 明确使用 `watchlist_` 前缀，避免与真实交易系统混淆

### 15.2 动作枚举建议

- `WATCH`：仅观察，不建议试仓
- `PROBE`：允许试仓，轻仓验证
- `ADD`：趋势/结构较强，可考虑加仓
- `HOLD`：继续持有，暂不调整
- `TRIM`：减仓控制风险
- `EXIT`：退出观察或持仓

### 15.3 风险提示建议

第一批固定风险标签建议：

- `HIGH_VOLATILITY`
- `OVERBOUGHT`
- `OVERSOLD`
- `TREND_WEAKENING`
- `HIGH_CORRELATION`
- `CROWDING_RISK`
- `EVENT_UNCERTAINTY`

### 15.4 LLM 使用边界

LLM 可以做：

- 综合解释
- 给出摘要
- 生成用户可读建议
- 对多个建议排序
- 在固定 schema 内填充结构化解释字段

LLM 不可以做：

- 原始指标计算
- 仓位上限硬约束计算
- 替代规则引擎做基础技术判定
- 生成未定义 schema 的随意字段

### 15.5 默认行为冻结

- 阶段一 `TimingAnalysisCard.actionBias` 在无持仓上下文时只允许产出 `WATCH | PROBE | ADD`
- `HOLD | TRIM | EXIT` 仅在存在 `PortfolioSnapshot` 或持仓上下文时允许出现
- 阶段二的“高相关性”先按行业/概念重叠与持仓集中度做规则提示，不引入滚动相关系数矩阵
- `TimingPreset` v1 只影响指标、agent 权重与阈值，不改变基础指标公式

### 15.6 测试与验证建议

Python 合约测试：

- 标准包装
- 参数校验
- 批量部分失败
- 标准错误体

TS 单元测试：

- timing value objects
- timing policies
- repository 映射
- reminder 目标约束

Workflow 测试：

- 模板码注册
- start 命令幂等
- 节点输出 schema 校验
- worker 恢复执行

集成测试：

- 单股 pipeline 成功落 `TimingSignalSnapshot` 与 `TimingAnalysisCard`
- watchlist 批量卡 pipeline 可为多个标的落表
- 阶段二在 `RISK_OFF` 下输出保守建议
- screening session 完成后能自动 enqueue timing pipeline
- 到期 review 能生成 `TimingReviewRecord` 并创建或消费 reminder

前端验收：

- run 页面展示 timing workflow 执行过程
- `/timing` 页面只读取持久化结果，不依赖内存态回传

## 18. 最终判断

`timing` 是本项目非常适合扩展的新上下文，且与现有 `screening / watchlist / workflow` 有天然协同关系。最合理的推进方式不是把项目改造成交易系统，而是先把它做成：

> 基于确定性技术指标、多 agent 协作、带有风险预算与复查能力的择时研究工作台。

本蓝图中的阶段一与阶段二适合当前项目用较高效率完成；阶段三则用于把 `timing` 从“建议生成器”升级为“研究闭环系统”。

与初版蓝图相比，本文档已补足以下 coding agent 必需信息：

- 写入口与读入口的职责边界
- Python 标准化 contract 与默认参数
- Prisma 关系策略与 reminder 泛化方向
- workflow 模板码、接入点与默认幂等键
- 各 agent 节点最小输入/输出 schema

因此，后续 coding agent 可以据此直接进入实现，不再需要自行决定上述高影响设计问题。
